Prompt Lab: URL Shortener

Run the prompts. Reflect. Then reveal why.

The Problem

Build a link shortener like bit.ly.

Small enough that everyone understands it in one sentence — so the focus stays on how you prompt, not on decoding the domain. Real requirements ambiguity, genuine design choices, a bounded build, and obvious tests.

Source: Classic system-design / coding-interview problem, widely used across interview-prep material. Adapted here as a small, re-runnable drill — no single canonical origin.

How This Lab Works

  • Open your AI tool (ChatGPT, Claude, Cursor…) in another window
  • Each round: run the ❌ poor prompt in a fresh chat, then the ✅ excellent prompt in another fresh chat
  • Reflect on the amber questions before moving on — compare what each version gave you
  • Then reveal the analysis and discuss it together

The point isn't the "right answer" on the slide — it's what you notice in the two outputs. Copy buttons put each prompt on your clipboard.

Round 1 · Requirements & Design — Scoping
Step 1 — Run the poor prompt (fresh chat)
❌ Poor prompt
"Write me a URL shortener."
🤔 Did it ask you anything, or just dump a generic design? Could you defend every choice it made — storage, ID scheme, scale?
Step 2 — Run the excellent prompt (fresh chat)
✅ Excellent prompt
"We're designing a URL shortener. Before any code, list the open requirements questions that most change the design — custom aliases, link expiration, click analytics, auth, and expected scale. Group them by how much they affect the architecture."
🤔 What decisions did it surface that you hadn't considered? Which would you defer to keep this solvable in an interview?

Why the poor prompt fails: One-shot outsourcing. You've surrendered every decision — scope, storage, ID scheme — to the AI. Violates Active Direction and Narrative Control.

Why the excellent prompt works: You use the AI to surface the decision space, then you make the calls. Ranking by architectural impact is Narrative Control — steering toward something completable in the time you have.

🗣️ Say out loud: "I'm scoping this to a single-instance, in-memory service with optional expiry, and deferring analytics and auth. That keeps it solvable in our time box — I'll note the deferred items as future work."

Round 2 · Requirements & Design — Short Codes
Step 1 — Run the poor prompt (fresh chat)
❌ Poor prompt
"What's the best way to generate the short codes?"
🤔 Did it hand you one answer as "the best"? Did it mention tradeoffs unprompted, or did you just accept it?
Step 2 — Run the excellent prompt (fresh chat)
✅ Excellent prompt
"Compare three short-code strategies — random string, hash of the URL, and a base62-encoded counter — on collision risk, guessability, and whether identical URLs collapse to one code. I'm leaning toward base62 counter; tell me where that choice hurts."
🤔 Did the comparison change your leaning? What did it say hurts about base62 — and would you have caught that yourself?

Why the poor prompt fails: "Best" invites an unexamined verdict you'll accept at face value. Violates Critical Reading.

Why the excellent prompt works: You ask for a comparison against criteria you named, state your leaning, and invite the AI to attack it — Collaboration + Critical Reading. You're pressure-testing a decision, not shopping for one.

🗣️ Say out loud: "Base62 counter gives me short, collision-free codes with trivial logic. The tradeoff is guessability and that it leaks volume — acceptable here, and I'd revisit it if links were sensitive."

Round 3 · Implementation — Build in Slices
Step 1 — Run the poor prompt (fresh chat)
❌ Poor prompt
"Now write all the code."
🤔 How many lines came back? Could you explain every one — and defend it if an interviewer asked "why this?"
Step 2 — Run the excellent prompt (fresh chat)
✅ Excellent prompt
"Let's build it in slices. Start with just the two core functions: shorten(url) returning a code, and resolve(code) returning the URL or a not-found signal. Use an in-memory dict and a base62 counter. No web layer yet — I want to eyeball the core logic first."
🤔 How much code came back this time? Could you review it in under a minute? Which version could you actually walk someone through?

Why the poor prompt fails: You get several hundred lines you didn't shape and can't defend. When the interviewer asks "why this?", you're reading your own solution for the first time.

Why the excellent prompt works: You delegate a well-bounded slice while keeping the design yours. You can actually read ~20 lines before moving on — Active Direction in practice.

🗣️ Say out loud: "I'm reviewing the counter increment for a race condition — single-threaded for now, so fine, but I'm flagging it as the first thing that breaks under concurrency."

Round 4 · Implementation — Extending
Step 1 — Run the poor prompt (continue the Step-2 chat)
❌ Poor prompt
"Add expiration too."
🤔 Did the AI guess what "expiration" means, or assume an implementation? What did it decide for you?
Step 2 — Run the excellent prompt instead
✅ Excellent prompt
"Add optional expiry: shorten(url, ttl_seconds=None). On resolve, treat an expired code as not-found and don't auto-delete — lazy expiry is fine. Keep the not-found path identical to a missing code so we don't leak whether a code ever existed."
🤔 Did it honor the "don't leak whether a code existed" requirement? Would you have thought to specify that?

Why the poor prompt fails: Vague bolt-on. You haven't said what expiry means here, so the AI guesses.

Why the excellent prompt works: You specify the behavior, the interface, and a security nuance the AI wouldn't infer. You're using it as a fast implementer of your decisions.

Round 5 · Testing
Step 1 — Run the poor prompt (fresh chat)
❌ Poor prompt
"Write some tests."
🤔 Are the tests happy-path only? What risky case did they skip entirely?
Step 2 — Run the excellent prompt (fresh chat)
✅ Excellent prompt
"List the edge cases worth testing, ranked by likelihood of hiding a bug — collisions, resolving an unknown code, an expired code, an empty or malformed URL, and idempotency of shortening the same URL twice. Then write tests for the top four."
🤔 Which edge case would you predict fails first? Did the AI's ranking match your instinct?

Why the poor prompt fails: You get happy-path tests that prove nothing interesting, and you've signaled you don't know what's risky. Violates Critical Reading and Articulation.

Why the excellent prompt works: You make the AI enumerate risk first, then you prioritize — you decide what "done" means. Narrative Control + Critical Reading.

🗣️ Say out loud: "The expired-code test is the one I most expect to fail. If the not-found responses for 'expired' and 'never existed' differ, that's a leak and a bug."

Debrief

Same Shape, Every Round

  • You surface options
  • You decide
  • The AI executes
  • You verify
🤔 Group discussion: In which round was the gap between poor and excellent the widest? Where did the poor prompt fool you into thinking it was "good enough"?

That loop — surface, decide, execute, verify — is the entire strategy in miniature, and it's exactly what the interviewer rubric scores.
